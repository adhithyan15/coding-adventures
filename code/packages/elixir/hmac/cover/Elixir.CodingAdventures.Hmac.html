3 HMAC (Hash-based Message Authentication Code) — RFC 2104 / FIPS 198-1.
4
5 ## What Is HMAC?
6
7 HMAC takes a secret key and a message and produces a fixed-size authentication
8 tag. It is used to verify BOTH the integrity (the message hasn't changed) AND
9 the authenticity (the sender knows the secret key) of a message.
10
11 HMAC is at the heart of:
12 - TLS 1.2 handshakes (HMAC-SHA256 for PRF)
13 - JWT (JSON Web Token) signatures (`HS256`, `HS512`)
14 - WPA2 Wi-Fi authentication (HMAC-SHA1 in PBKDF2)
15 - TOTP / HOTP one-time passwords (RFC 6238, RFC 4226)
16 - SSH MAC validation
17 - S3 / AWS Signature Version 4 request signing
18
19 ## Why Not hash(key || message)?
20
21 A naive approach would be `hash(key || message)`. This is vulnerable to a
22 **length extension attack**: anyone who knows the hash of a message can compute
23 the hash of a longer message without knowing the key.
24
25 Example with SHA-256 (a Merkle-Damgård construction):
26 ```
27 known: SHA-256(key || "GET /admin")
28 attacker can compute: SHA-256(key || "GET /admin" || padding || "/delete")
29 without knowing `key`
30 ```
31
32 This works because Merkle-Damgård hashes resume from the state left by the
33 previous block — if you know the state after hashing `key || message`, you can
34 just add more blocks.
35
36 ## The HMAC Construction (RFC 2104)
37
38 HMAC defeats length extension by using two nested hash calls with different keys:
39
40 ```
41 HMAC(K, M) = H((K' XOR opad) || H((K' XOR ipad) || M))
42
43 where:
44 K' = key padded (or hashed) to the hash's block size
45 ipad = 0x36 repeated to block size ("inner pad")
46 opad = 0x5C repeated to block size ("outer pad")
47 H = the underlying hash function (MD5, SHA-1, SHA-256, SHA-512)
48 || = concatenation
49 ```
50
51 **Why two nested calls work:**
52
53 The inner call `H(ipad_key || M)` produces a hash that commits to the message
54 under a modified key. The outer call `H(opad_key || inner)` wraps that result
55 under a DIFFERENT modified key. An attacker cannot extend the message because
56 they would need to break into the outer hash, which they cannot do without
57 knowing `opad_key`.
58
59 **The constants 0x36 and 0x5C:**
60
61 These values were chosen by Hugo Krawczyk (the main HMAC designer) because
62 they are Hamming-distance-maximized hex values: 0x36 = 0011_0110 and
63 0x5C = 0101_1100. They differ in 4 of 8 bits, ensuring the inner and outer
64 keys are as different as possible even though both are derived from K'.
65
66 ## Step-by-Step Algorithm
67
68 ```
69 Input: key K (any length), message M, hash function H, block size B
70
71 Step 1 — Normalize the key to exactly B bytes:
72 If len(K) > B: K' = H(K) (hash long keys down to digest size)
73 If len(K) < B: K' = K || 0x00 * (B - len(K)) (zero-pad short keys)
74 If len(K) == B: K' = K
75
76 Step 2 — Create padded keys:
77 inner_key = K' XOR (0x36 * B) (every byte of K' XOR'd with 0x36)
78 outer_key = K' XOR (0x5C * B) (every byte of K' XOR'd with 0x5C)
79
80 Step 3 — Compute the nested hash:
81 inner = H(inner_key || M)
82 result = H(outer_key || inner)
83
84 Output: result (same length as H's digest)
85 ```
86
87 ## Block Sizes and Digest Sizes
88
89 | Algorithm | Block Size | Digest Size | RFC Reference |
90 |-----------|-----------|-------------|---------------|
91 | MD5 | 64 bytes | 16 bytes | RFC 1321 |
92 | SHA-1 | 64 bytes | 20 bytes | FIPS 180-4 |
93 | SHA-256 | 64 bytes | 32 bytes | FIPS 180-4 |
94 | SHA-512 | 128 bytes | 64 bytes | FIPS 180-4 |
95
96 SHA-512 uses a 128-byte block because it processes 64-bit words (not 32-bit),
97 so it needs twice as many bytes per block to hold the same number of words.
98 This means the ipad and opad strings are 128 bytes long, not 64.
99

287 # normalize_key/3 — bring any key to exactly block_size bytes.
288 #
289 # Three cases from RFC 2104 Section 2:
290 # len(key) > block_size → hash the key, then zero-pad the hash to block_size
291 # len(key) < block_size → zero-pad the key on the right
292 # len(key) == block_size → use the key as-is
293 #
294 # Note: hashing a long key may produce a digest shorter than block_size
295 # (e.g. SHA-256 produces 32 bytes; for block_size=64 we then zero-pad).
296 defp normalize_key(hash_fn, block_size, key) do
297 59 key_len = byte_size(key)
298
299 59 normalized =
300 if key_len > block_size do
301 10 hash_fn.(key)
302 else
303 49 key
304 end
305
306 # Zero-pad to exactly block_size
307 59 pad_size = block_size - byte_size(normalized)
308
309 59 if pad_size > 0 do
310 57 normalized <> :binary.copy(<<0x00>>, pad_size)
311 else
312 2 normalized
313 end
314 end
