# Ed25519 — Edwards-Curve Digital Signature Algorithm

## Overview

Ed25519 (RFC 8032) is a digital signature scheme using the Edwards
form of Curve25519. Designed by Daniel J. Bernstein, Niels Duif, Tanja
Lange, Peter Schwabe, and Bo-Yin Yang, it provides 128-bit security
with extremely fast signing and verification.

Used in: SSH (since OpenSSH 6.5), TLS 1.3, Signal Protocol, Tor,
GnuPG, WireGuard, and most modern cryptocurrency systems.

### Why Ed25519?

- **Fast:** ~70,000 signatures/second on a modern CPU. Verification
  is batch-friendly.
- **Compact:** 32-byte public keys, 64-byte signatures — half the
  size of RSA-2048.
- **Deterministic signing:** No random nonce needed. The "nonce" is
  derived from the private key and message via hashing. This
  eliminates an entire class of catastrophic failures (Sony PlayStation
  3 ECDSA disaster, where nonce reuse leaked the private key).
- **Side-channel resistant:** The Edwards curve formulas use complete
  addition (no special cases for point doubling or identity), and the
  scalar multiplication can be implemented in constant time.
- **No cofactor headaches (mostly):** Ed25519 uses a cofactor-8 curve,
  but the standard signing/verification procedure handles this cleanly.

### The Curve

Ed25519 operates on a twisted Edwards curve:

```
-x^2 + y^2 = 1 + d * x^2 * y^2

where d = -121665/121666 mod p
      p = 2^255 - 19       (the base field prime)
```

This curve is birationally equivalent to Curve25519 (the Montgomery
form used in X25519 key exchange). The group of rational points has
order `8 * L` where:

```
L = 2^252 + 27742317777372353535851937790883648493
```

L is a 253-bit prime. The base point B has order L (it generates the
prime-order subgroup).

### Base Point

```
B_y = 4/5 mod p
    = 46316835694926478169428394003475163141307993866256225615783033890098355573289
B_x = recovered from the curve equation (positive square root)
    = 15112221349535807912866137220509078750507884956996801397894590994520548966982
```

## Algorithm

### Key Generation

```
Input: 32-byte random seed (the "private key seed")

1. h = SHA-512(seed)                           // 64 bytes
2. a = h[0..31]                                // first 32 bytes
   Clamp a:
     a[0]  &= 248      // clear lowest 3 bits (cofactor)
     a[31] &= 127      // clear highest bit
     a[31] |= 64       // set second-highest bit
   Now a is the secret scalar.
3. A = a * B                                   // public key (point)
4. Encode A as 32 bytes:
   - Take the y-coordinate as 256-bit little-endian
   - Set the high bit of the last byte to the low bit of x
   This is the 32-byte public key.

Output:
  public_key = encoded A (32 bytes)
  secret_key = seed || public_key (64 bytes)
```

The 64-byte secret key format (seed + public key) is the standard
representation. The actual signing scalar and prefix are derived on
the fly from the seed via SHA-512.

### Signing

```
Input: message M, secret_key (seed || public_key)

1. h = SHA-512(seed)                           // 64 bytes
2. a = clamp(h[0..31])                         // secret scalar
3. prefix = h[32..63]                          // second half
4. r = SHA-512(prefix || M) mod L              // deterministic nonce
5. R = r * B                                   // nonce point
6. Encode R as 32 bytes
7. S = (r + SHA-512(R || A || M) * a) mod L    // signature scalar
8. Encode S as 32-byte little-endian integer

Output: signature = R || S (64 bytes)
```

The nonce r is deterministic — derived from the message and a secret
prefix. This means signing the same message twice produces the same
signature (desirable!) and eliminates the catastrophic risk of a bad
random number generator.

### Verification

```
Input: message M, signature (R || S), public_key A

1. Decode R as a curve point (reject if not on curve)
2. Decode S as a little-endian integer (reject if S >= L)
3. Decode A as a curve point (reject if not on curve)
4. k = SHA-512(R || A || M) mod L
5. Check: 8 * S * B == 8 * R + 8 * k * A

Output: true if the equation holds, false otherwise
```

The factor of 8 (the cofactor) ensures that small-subgroup components
cancel out. Some implementations check `S * B == R + k * A` instead
(without the cofactor multiplication), which is also acceptable for
standard Ed25519.

### Point Arithmetic on Edwards Curves

Addition of points `(x1, y1)` and `(x2, y2)`:

```
x3 = (x1*y2 + y1*x2) / (1 + d*x1*x2*y1*y2)
y3 = (y1*y2 + x1*x2) / (1 - d*x1*x2*y1*y2)
```

In projective (extended) coordinates `(X, Y, Z, T)` where `x = X/Z`,
`y = Y/Z`, `T = X*Y/Z`, the formulas use only field multiplications
and additions — no divisions until the final conversion back to
affine coordinates.

The key insight: these formulas are *complete* — they work for all
input points including doubling and the identity point, with no special
cases. This makes constant-time implementations straightforward.

### Field Arithmetic (mod p)

All operations are modulo `p = 2^255 - 19`. This prime is chosen for
speed: reduction mod p only requires multiplication by 19 and
addition, since `2^255 ≡ 19 (mod p)`.

Key operations:
- **Addition/subtraction:** Standard, then conditional subtract/add p.
- **Multiplication:** Schoolbook or Karatsuba on 5×51-bit or 10×25.5-bit
  limbs, then reduce.
- **Inversion:** Via Fermat's little theorem: `a^(-1) = a^(p-2) mod p`.
- **Square root:** `a^((p+3)/8) mod p` (since `p ≡ 5 mod 8`), with a
  correction factor of `sqrt(-1)` if needed.

## Interface Contract

| Function | Signature | Description |
|----------|-----------|-------------|
| `generate_keypair` | `(seed_32) -> (public_key_32, secret_key_64)` | Generate key pair from 32-byte seed. |
| `sign` | `(message, secret_key_64) -> signature_64` | Deterministic signature. |
| `verify` | `(message, signature_64, public_key_32) -> bool` | Verify a signature. Returns false on any error. |

Internal functions (must be implemented, may not be public API):
- `scalar_mult(scalar, point) -> point` — Constant-time scalar multiplication.
- `point_add(P, Q) -> R` — Extended coordinate addition.
- `point_encode(point) -> 32 bytes` — Compress point to y-coordinate + sign bit.
- `point_decode(32 bytes) -> point` — Decompress and validate on-curve.
- `field_mul`, `field_add`, `field_inv` — Modular arithmetic in GF(p).
