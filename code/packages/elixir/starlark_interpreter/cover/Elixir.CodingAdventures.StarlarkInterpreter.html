5 ## Chapter 1: What Is an Interpreter?
6
7 An interpreter takes source code and executes it. Unlike a compiler that
8 produces an executable file, an interpreter runs the program directly. Our
9 Starlark interpreter uses a **multi-stage pipeline** internally:
10
11 source code -> tokens -> AST -> bytecode -> execution
12
13 Each stage is handled by a separate package:
14
15 1. **Lexer** (built into the compiler): Breaks source text into tokens.
16 `"x = 1 + 2"` -> `[NAME("x"), EQUALS, INT("1"), PLUS, INT("2")]`
17
18 2. **Parser** (built into the compiler): Groups tokens into an AST.
19 `[NAME, EQUALS, INT, PLUS, INT]` -> `AssignStmt(x, Add(1, 2))`
20
21 3. **Compiler** (starlark_ast_to_bytecode_compiler): Translates AST into
22 bytecode. `AssignStmt(x, Add(1, 2))` ->
23 `[LOAD_CONST 1, LOAD_CONST 2, ADD, STORE_NAME x]`
24
25 4. **VM** (starlark_vm): Executes bytecode on a virtual stack machine.
26
27 This package chains them together and adds the critical `load()` function.
28
29 ## Chapter 2: The load() Function
30
31 `load()` is what makes BUILD files work. It's how a BUILD file imports
32 rule definitions from a shared library:
33
34 load("//rules/python.star", "py_library")
35
36 py_library(
37 name = "mylib",
38 deps = ["//other:lib"],
39 )
40
41 When the VM encounters a `load()` call:
42
43 1. **Resolve** the path — find the file contents
44 2. **Execute** the file through the same interpreter pipeline
45 3. **Extract** the requested symbols from the result
46 4. **Inject** them into the current scope
47
48 ## Chapter 3: Usage Examples
49
50 **Simple execution:**
51
52 result = StarlarkInterpreter.interpret("x = 1 + 2\\nprint(x)\\n")
53 result.variables["x"] #=> 3
54 result.output #=> ["3"]
55
56 **With load():**
57
58 files = %{
59 "//rules/math.star" => "def double(n):\\n return n * 2\\n"
60 }
61 result = StarlarkInterpreter.interpret(
62 "load(\\"//rules/math.star\\", \\"double\\")\\nresult = double(21)\\n",
63 file_resolver: files
64 )
65 result.variables["result"] #=> 42
66
67 **From a file:**
