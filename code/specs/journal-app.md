# Journal App — Markdown Journal with Pluggable Storage

## Overview

Most journaling apps lock you into their proprietary format or require a
cloud subscription. What if your journal entries were just markdown — the
same format you already use for READMEs, documentation, and notes?

This app is a local-first markdown journal. You write in GitHub Flavored
Markdown, see a live rendered preview, and your entries are stored locally
via a pluggable storage backend. Today that backend is IndexedDB in the
browser. Tomorrow it could be Google Drive, SQLite, or a flat file on disk.
The app never knows the difference — it talks to a `Storage` interface and
the backend handles the rest.

Entries are dated and organized by creation date. You can write multiple
entries per day. The editor is a split pane: raw markdown on the left,
rendered HTML preview on the right, updating live as you type.

The app ships as both a web app (deployed to GitHub Pages) and an Electron
desktop app (packaged for macOS, Windows, and Linux). The codebase is
identical — Electron wraps the same React app that runs in the browser.

The V1 goal is to explore the full integration of the repo's existing
packages: `@coding-adventures/storage` for persistence abstraction,
`@coding-adventures/gfm` for markdown rendering, `@coding-adventures/store`
for Flux state management, and `@coding-adventures/ui-components` for
shared theming and UI primitives.

## Core Concepts

### Entry

An **Entry** is one journal entry. It contains a title, a body written in
GitHub Flavored Markdown, and timestamps. Multiple entries can exist on
the same calendar date.

The `createdAt` field is an ISO 8601 date string (`"YYYY-MM-DD"`) — not a
timestamp — because a journal entry belongs to a calendar date, not a
point in time. "I wrote this on March 15th" means the same thing regardless
of timezone. This also enables simple string comparison for date sorting
and grouping: `"2026-03-15" < "2026-04-02"`.

The `updatedAt` field is a Unix timestamp (milliseconds) because edit times
do care about precision — knowing whether two edits happened seconds apart
matters for conflict resolution in future sync backends.

### Timeline

The **Timeline** is the home screen. It shows all entries grouped by date,
in reverse chronological order (newest first). Each date heading expands
to show the entries created that day.

### Split-Pane Editor

The editor renders two panes side by side:
- **Left pane**: a `<textarea>` for writing raw GFM markdown
- **Right pane**: rendered HTML preview, updated live as you type

The preview is generated by piping the markdown through the GFM pipeline:
`parse()` → `sanitize()` → `toHtml()`. Preview updates are debounced
(150ms) to avoid thrashing on fast typists.

```
│                                 │                       │
│                      ┌──────────▼───────────────────┐   │
│                      │   Storage Interface           │   │
│                      │   (from @c-a/storage)         │   │
│                      │                              │   │
│                      │   open() / close()           │   │
│                      │   get() / getAll()           │   │
│                      │   put() / delete()           │   │
│                      │   query() / transaction()    │   │
│                      └──────────┬───────────────────┘   │
│                                 │                       │
│              ┌──────────────────┼──────────────────┐    │
│              ▼                  ▼                  ▼    │
│  ┌───────────────┐  ┌──────────────┐  ┌────────────┐   │
│  │ IndexedDB     │  │ MemoryStorage│  │ Future:    │   │
│  │ Storage       │  │ (tests)      │  │ Google     │   │
│  │ (default)     │  │              │  │ Drive,     │   │
│  │               │  │              │  │ SQLite,    │   │
│  │               │  │              │  │ S3, ...    │   │
│  └───────────────┘  └──────────────┘  └────────────┘   │
└─────────────────────────────────────────────────────────┘
```

The app imports `Storage` from `@coding-adventures/storage` — the unified
storage interface that abstracts over all backends. It never imports
`IndexedDBStorage` directly except during initialization, where the
concrete backend is selected and injected into the persistence middleware.

This design means swapping to Google Drive storage requires:
1. Writing a `GoogleDriveStorage` class that implements the `Storage`
   interface
2. Changing the initialization code in `main.tsx` to instantiate it
3. Zero changes to the store, reducer, actions, persistence middleware,
   or any UI component

### Storage Schema

```typescript
const STORAGE_CONFIG: StorageConfig = {
  dbName: "journal-app",
  version: 1,
  stores: [
    {
      name: "entries",
      keyPath: "id",
      indexes: [
        { name: "createdAt", keyPath: "createdAt" },
      ],
    },
  ],
};
```

One store, one index. The `createdAt` index enables efficient querying by
date when the backend supports native indexes (IndexedDB, SQLite). For
backends that don't (Google Drive, flat files), the sql-execution-engine
handles filtering in memory.

## State Management

### Store

The store follows the same Flux architecture as the checklist-app and
Engram:

```
User action → store.dispatch(action) → middleware chain → reducer → new state
                                            │
                                    persistence middleware
                                    writes to Storage (fire-and-forget)
                                            │
                                    store emits "change"
                                            │
                                    useStore() re-renders components
```

The store is a singleton created in `state.ts`:

```typescript
import { Store } from "@coding-adventures/store";
import { reducer } from "./reducer.js";
import type { AppState } from "./reducer.js";

const initialState: AppState = { entries: [] };

export const store = new Store<AppState>(initialState, reducer);
```

### Actions

```typescript
// Action type constants
const ENTRY_CREATE   = "ENTRY_CREATE";
const ENTRY_UPDATE   = "ENTRY_UPDATE";
const ENTRY_DELETE   = "ENTRY_DELETE";
const ENTRIES_LOAD   = "ENTRIES_LOAD";

// Action creators

function entryCreateAction(title: string, content: string): Action
// → { type: ENTRY_CREATE, title, content }
// Reducer generates id (UUID), createdAt (today's date), updatedAt (now)

function entryUpdateAction(
  id: string,
  title: string,
  content: string,
): Action
// → { type: ENTRY_UPDATE, id, title, content }
// Reducer sets updatedAt to now

function entryDeleteAction(id: string): Action
// → { type: ENTRY_DELETE, id }

function entriesLoadAction(entries: Entry[]): Action
// → { type: ENTRIES_LOAD, entries }
// Used during initialization to hydrate state from storage
```

### Reducer

The reducer is a pure function: `(AppState, Action) → AppState`.

```
ENTRY_CREATE:
  - Generate id = crypto.randomUUID()
  - Set createdAt = new Date().toISOString().slice(0, 10)  // "YYYY-MM-DD"
  - Set updatedAt = Date.now()
  - Append new Entry to state.entries

ENTRY_UPDATE:
  - Find entry by id
  - Replace title and content
  - Set updatedAt = Date.now()
  - Return new state with updated entry

ENTRY_DELETE:
  - Filter out entry with matching id
  - Return new state without the entry

ENTRIES_LOAD:
  - Replace state.entries with the loaded entries array
```

### Persistence Middleware

The persistence middleware takes a `Storage` instance (not
`IndexedDBStorage` — the interface) and maps actions to storage writes:

```
ENTRY_CREATE   → storage.put("entries", newEntry)
ENTRY_UPDATE   → storage.put("entries", updatedEntry)
ENTRY_DELETE   → storage.delete("entries", id)
ENTRIES_LOAD   → no-op (data came FROM storage)
```

All writes are fire-and-forget: the middleware calls `storage.put()` or
`storage.delete()` without awaiting the Promise. This keeps the UI
responsive — dispatch returns immediately after the reducer runs.

Each entry card shows:

- **New Entry** button → navigates to `#/entry/new`
- **Click an entry card** → navigates to `#/entry/:id` (view mode)

Date headings are computed by grouping entries on `createdAt`. The grouping
algorithm:

```typescript
function groupByDate(entries: Entry[]): Map<string, Entry[]> {
  const groups = new Map<string, Entry[]>();
  for (const entry of entries) {
    const existing = groups.get(entry.createdAt) ?? [];
    existing.push(entry);
    groups.set(entry.createdAt, existing);
  }
  return groups;
}
```

Dates are sorted descending. Entries within each date are sorted by
`updatedAt` descending (most recently edited first).

```
│  └────────────────────┴────────────────────────┘   │
└─────────────────────────────────────────────────────┘
```

**Left pane**: `<textarea>` with monospace font, full height. The user types
raw GFM markdown here.

**Right pane**: Read-only HTML preview. Rendered by the GFM pipeline:

```typescript
import { parse, toHtml } from "@coding-adventures/gfm";
import { sanitize, RELAXED } from "@coding-adventures/document-ast-sanitizer";

function renderPreview(markdown: string): string {
  const ast = parse(markdown);
  const sanitized = sanitize(ast, RELAXED);
  return toHtml(sanitized);
}
```

The RELAXED sanitization policy is appropriate because this is the user's
own content — not untrusted user-generated input. RELAXED allows HTML pass-
through (for embedding) and permits `http`, `https`, `mailto`, and `ftp`
URL schemes.

**Live preview**: The preview updates on every keystroke, debounced at 150ms.
This means if the user types continuously, the preview re-renders at most
~6.7 times per second — fast enough to feel instant, slow enough to avoid
jank.

```typescript
const DEBOUNCE_MS = 150;
let debounceTimer: number | null = null;

function handleContentChange(value: string) {
  setContent(value);
  if (debounceTimer !== null) clearTimeout(debounceTimer);
  debounceTimer = window.setTimeout(() => {
    setPreviewHtml(renderPreview(value));
  }, DEBOUNCE_MS);
}
```

Actions:
- **Save** → dispatches `ENTRY_CREATE`, navigates to `#/`
- **Back** → navigates to `#/` (discards unsaved content)

## GFM Integration

The markdown rendering pipeline uses three existing packages:

```
Raw markdown string
    │
    ▼
parse() from @coding-adventures/gfm
    │  Parses GFM into a DocumentNode AST.
    │  Supports: headings, paragraphs, lists, code blocks,
    │  blockquotes, tables, emphasis, strong, links, images,
    │  strikethrough, task lists, autolinks, hard/soft breaks.
    ▼
sanitize() from @coding-adventures/document-ast-sanitizer
    │  Applies RELAXED policy:
    │  - Allows HTML raw blocks (format: "html")
    │  - Allows http, https, mailto, ftp URL schemes
    │  - Images pass through unchanged
    │  - Headings unrestricted (h1–h6)
    ▼
toHtml() from @coding-adventures/document-ast-to-html
    │  Renders the sanitized AST to an HTML string.
    │  Handles: tight vs loose lists, code block language
    │  classes, table alignment, task list checkboxes.
    ▼
HTML string → injected into preview pane via dangerouslySetInnerHTML
```

### Preview Styling

The preview pane needs typography CSS for the rendered HTML. The stylesheet

## Startup Flow

```
1. initI18n(en)
2. Try to open IndexedDBStorage(STORAGE_CONFIG)
   └─ on failure: fall back to MemoryStorage
3. entries = await storage.getAll<Entry>("entries")
4. store.use(createPersistenceMiddleware(storage))
5. if (entries.length > 0)
     dispatch(entriesLoadAction(entries))
   else
     dispatch seed entries (optional — see Seed Data below)
6. createRoot(root).render(<App />)
```

The storage backend is selected once during initialization. The persistence
middleware receives the `Storage` instance and uses it for all subsequent
writes. No other code in the app knows or cares which backend is active.

## Test Strategy

### Unit tests — `src/__tests__/reducer.test.ts`

Pure logic in the reducer tested in isolation (Node environment, no DOM).
Target: 95%+ line coverage.

Required test cases:

- `ENTRY_CREATE` generates a unique id, correct createdAt date, updatedAt
- `ENTRY_CREATE` appends to existing entries (does not replace)
- `ENTRY_UPDATE` updates title, content, and updatedAt
- `ENTRY_UPDATE` does not modify other entries
- `ENTRY_UPDATE` with unknown id returns state unchanged
- `ENTRY_DELETE` removes the entry with matching id
- `ENTRY_DELETE` with unknown id returns state unchanged
- `ENTRIES_LOAD` replaces entries array entirely
- `ENTRIES_LOAD` with empty array clears all entries
- Reducer returns state unchanged for unknown action types
- All state transitions are immutable (new objects, not mutations)

## V1 Scope

**In scope:**
- Entry CRUD (create, read, update, delete)
- GFM markdown rendering with live preview
- Split-pane editor
- Timeline view grouped by date
- Pluggable storage via `Storage` interface
- Default IndexedDB backend with MemoryStorage fallback
- Seed data on first visit
- Hash-based router (4 routes, 3 screens + 1 shared component)
- Electron desktop app packaging
- GitHub Actions release workflow
- GitHub Pages web deploy
- i18n with `@coding-adventures/ui-components`
- Dark theme via shared `theme.css`
- Unit tests for all state logic; component tests with RTL

**Out of scope for V1:**
- Full-text search
- Tags or categories
- Export (markdown files, PDF)
- Encryption
- Sync between devices
- Google Drive or other cloud storage backends (interface is ready, implementation deferred)
- Entry attachments (images, files)
- Markdown toolbar (bold, italic, heading buttons)
- Mobile-responsive layout
- Drag-and-drop reordering
- Undo / redo
- Dark/light theme toggle (dark only, via theme.css)

## Future Extensions

- **V2 — Cloud Sync**: Implement `GoogleDriveStorage` backend. Add storage
  selector in settings. Conflict resolution via last-write-wins on
  `updatedAt` timestamp.
- **V3 — Search**: Full-text search across entry titles and content using
  the sql-execution-engine's `LIKE` operator or a dedicated search index.
- **V4 — Tags**: Tag entries with labels, filter timeline by tag, tag
  auto-complete.
- **V5 — Export**: Download entries as `.md` files. Bulk export as a zip
  archive with one file per entry, named `YYYY-MM-DD-title.md`.
- **V6 — Encryption**: Client-side encryption layer (PBKDF2 + AES-GCM)
  between the store and the storage backend. Entries encrypted at rest
  regardless of backend.
