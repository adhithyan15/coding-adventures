# MA01 — GF(256): Galois Field GF(2^8) Arithmetic

## Overview

**GF(2^8)** — read as "GF of 2 to the 8th" or "GF of 256" — is a finite field
with exactly 256 elements. The elements are the integers 0 through 255, but the
arithmetic is very different from ordinary integer arithmetic.

### Why Finite Fields?

Three important algorithms in modern computing rely on GF(256):

1. **Reed-Solomon error correction** — Used in QR codes, CDs, DVDs, hard drives,
   and deep-space communication. RS codes treat data bytes as field elements and
   perform polynomial arithmetic over GF(256) to add redundancy that can detect and
   correct burst errors.

2. **QR codes** — The error correction codewords in a QR code are a Reed-Solomon
   code over GF(256). A QR code can survive up to 30% damage thanks to this.

3. **AES encryption** — The AES SubBytes step (the S-box) and the MixColumns step
   use arithmetic in GF(2^8) with the AES polynomial `x^8 + x^4 + x^3 + x + 1`.

This package implements GF(256) with the **Reed-Solomon primitive polynomial**
`x^8 + x^4 + x^3 + x^2 + 1` (= `0x11D`), which is the standard for most QR and
RS implementations.

### Characteristic 2 Means Add = XOR = Subtract

In **characteristic 2** fields, `1 + 1 = 0`. This sounds strange, but it is
consistent: if `1 + 1 = 0`, then `1 = -1`, so subtraction is the same as addition.

Concretely, for **bytes** (elements of GF(2^8)):

```
add(a, b) = a XOR b
subtract(a, b) = a XOR b        ← same operation!
```

This is because each bit of a byte is an element of GF(2) = {0, 1}, and in GF(2),
`1 + 1 = 0 mod 2`. The XOR of two bytes performs GF(2) addition on each bit
independently.

This means you never need a carry bit for addition in GF(256) — XOR is exact.

## The Primitive Polynomial

The elements of GF(2^8) are polynomials over GF(2) of degree ≤ 7:

```
a₇x⁷ + a₆x⁶ + a₅x⁵ + a₄x⁴ + a₃x³ + a₂x² + a₁x + a₀
```

where each `aᵢ ∈ {0, 1}`. This gives 2^8 = 256 elements.

To multiply two such polynomials, the product can have degree up to 14. We reduce
it modulo an **irreducible polynomial** of degree 8, just as integers modulo a prime
give a finite field.

We use the **primitive polynomial**:

```
p(x) = x^8 + x^4 + x^3 + x^2 + 1
```

In binary (bit 8 = x^8, bit 4 = x^4, bit 3 = x^3, bit 2 = x^2, bit 0 = x^0):

```
0b1_0001_1101 = 0x11D = 285
```

### Why Irreducible?

A polynomial over GF(2) is irreducible if it cannot be factored into two
polynomials of lower degree. We need irreducibility to ensure that division
is always possible (every non-zero element has a multiplicative inverse).
This is the same reason we use prime moduli for integer fields: `ℤ/pℤ` is a
field only when `p` is prime.

There are 30 irreducible polynomials of degree 8 over GF(2). We choose one that
is also **primitive** — meaning the element `g = 2 = x` (i.e., the polynomial `x`)
is a generator of the entire multiplicative group.

### What "Generates the Full Multiplicative Group" Means

The 255 non-zero elements of GF(256) form a **cyclic group** under multiplication.
A generator `g` satisfies:

```
g⁰ = 1,  g¹ = g,  g²,  g³,  ...,  g²⁵⁴,  g²⁵⁵ = 1  (cycles back)
```

Every non-zero field element appears exactly once in the list `g⁰, g¹, ..., g²⁵⁴`.

With our primitive polynomial `0x11D`, the element `g = 2` is such a generator.
This means we can represent multiplication using **logarithms**:

```
a × b = g^(log_g(a) + log_g(b))
```

which turns a multiplication into two table lookups and an addition modulo 255.
This is fast and avoids complex bit-level polynomial multiplication.

---

## Log/Antilog Table Construction

We precompute two lookup tables at startup:

- `ALOG[i]` = g^i mod p(x)   — the antilogarithm table, 255 entries
- `LOG[x]` = i such that g^i = x   — the logarithm table, 256 entries (LOG[0] is unused)

### Algorithm

Start with `g = 2`. Repeatedly multiply by `g` (shift left by 1 bit). If the
result overflows a byte (bit 8 is set), XOR with `0x11D` to reduce modulo `p(x)`.

```
ALOG[0] = 1
for i = 1 to 254:
    val = ALOG[i-1] * 2
    if val >= 256:   (bit 8 is set)
        val = val XOR 0x11D
    ALOG[i] = val
```

Simultaneously, invert: `LOG[ALOG[i]] = i`.

### First 10 Entries

| i | ALOG[i] = 2^i mod p | hex |
|---|---------------------|-----|
| 0 | 1                   | 0x01 |
| 1 | 2                   | 0x02 |
| 2 | 4                   | 0x04 |
| 3 | 8                   | 0x08 |
| 4 | 16                  | 0x10 |
| 5 | 32                  | 0x20 |
| 6 | 64                  | 0x40 |
| 7 | 128                 | 0x80 |
| 8 | 29  (256 XOR 0x11D = 256-285=-29? No: 256 XOR 285 = 0x100 XOR 0x11D = 0x01D = 29) | 0x1D |
| 9 | 58  (29 * 2 = 58, no overflow) | 0x3A |

At i=8: `128 * 2 = 256`. Since 256 ≥ 256, XOR with 0x11D = 285:
`256 XOR 285 = 0x100 XOR 0x11D = 0x01D = 29`. So `ALOG[8] = 29`.

### Last 5 Entries

| i | ALOG[i] | hex |
|---|---------|-----|
| 250 | 36  | 0x24 |
| 251 | 72  | 0x48 |
| 252 | 144 | 0x90 |
| 253 | 57  | 0x39 |
| 254 | 114 | 0x72 |

And the cycle: `ALOG[255]` would be `ALOG[0] = 1` again, confirming `g²⁵⁵ = 1`.

## Operations

### add(a, b) and subtract(a, b)

```
add(a, b) = a XOR b
subtract(a, b) = a XOR b
```

No tables needed. XOR is the only operation. Since characteristic is 2, negation
is identity: every element equals its own negation.

### multiply(a, b)

```
multiply(a, b):
    if a == 0 or b == 0:
        return 0
    return ALOG[(LOG[a] + LOG[b]) % 255]
```

The special case for zero is necessary because LOG[0] is undefined (there is no
power of a generator that equals zero).

**Example**: `0x03 × 0x07 = ?`
```
LOG[0x03] = ?   →  2 is g¹, so ... find 3 in ALOG: ALOG[25] = 3
LOG[0x07] = ?   →  find 7 in ALOG: need to compute...
```
(The exact log values depend on the table; the point is the method.)

### divide(a, b)

```
divide(a, b):
    if b == 0: ERROR (division by zero)
    if a == 0: return 0
    return ALOG[(LOG[a] - LOG[b] + 255) % 255]
```

The `+ 255` before `% 255` ensures the result is non-negative when `LOG[a] < LOG[b]`.

### power(base, exp)

```
power(base, exp):
    if base == 0:
        return 0 if exp > 0 else ERROR   (0^0 is undefined)
    return ALOG[(LOG[base] * exp) % 255]
```

Note: `exp` can be any non-negative integer; we reduce `LOG[base] * exp` modulo 255
because the multiplicative group has order 255.

### inverse(a)

The multiplicative inverse of `a` satisfies `a × inverse(a) = 1`.

```
inverse(a):
    if a == 0: ERROR (zero has no inverse)
    return ALOG[255 - LOG[a]]
```

Why? Because `a × ALOG[255 - LOG[a]] = ALOG[LOG[a] + 255 - LOG[a]] = ALOG[255] = 1`.

## Verification — GF(256) Spot Checks

These are well-known test vectors for GF(256) with the `0x11D` polynomial:

```
0x53 × 0x8C = 0x01
```
(0x53 is the multiplicative inverse of 0x8C under the 0x11D polynomial)

Note: The well-known pair `0x53 × 0xCA = 0x01` is specific to the **AES**
primitive polynomial `x^8 + x^4 + x^3 + x + 1 = 0x11B`. AES uses a different
polynomial than Reed-Solomon. With our 0x11D polynomial, `inverse(0x53) = 0x8C`.

```
g^255 = 1
```
(The generator has multiplicative order exactly 255)

```
x × inverse(x) = 1,   for all x in 1..255
```
(Every non-zero element has a multiplicative inverse)

```
add(x, x) = 0,   for all x
```
(Characteristic 2: every element is its own additive inverse)

```
multiply(x, 1) = x,   for all x
```
(1 is the multiplicative identity)

|----------|-----------|---------|--------|
| `add(a, b)` | `(GF256, GF256) → GF256` | a XOR b | — |
| `subtract(a, b)` | `(GF256, GF256) → GF256` | a XOR b | — |
| `multiply(a, b)` | `(GF256, GF256) → GF256` | product | — |
| `divide(a, b)` | `(GF256, GF256) → GF256` | quotient | b = 0 |
| `power(base, exp)` | `(GF256, int) → GF256` | base^exp | base=0 and exp≤0 |
| `inverse(a)` | `(GF256) → GF256` | a^(-1) | a = 0 |
| `zero()` | `() → GF256` | 0 | — |
| `one()` | `() → GF256` | 1 | — |

Constants:
- `PRIMITIVE_POLYNOMIAL = 0x11D`
- `LOG[256]` — logarithm table, LOG[0] is unused/0
- `ALOG[255]` — antilogarithm table
