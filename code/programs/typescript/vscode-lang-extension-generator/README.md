# vscode-lang-extension-generator

Generate VS Code extensions wrapping LSP and/or DAP servers for any
language built on the LANG-VM pipeline.  One command in, a fully wired
extension out — no boilerplate to copy-paste.

## Why this exists

Once a language has a working `*-lsp-server` (per LS02) and/or `*-dap`
(per LS03) binary, the only thing standing between it and a real
authoring experience inside VS Code is a thin **extension** — a few
hundred bytes of `package.json` declaring contributions, plus an
`extension.ts` activation function that launches the appropriate
subprocesses.

Hand-writing this scaffold per language has the same problem the
monorepo's other scaffold-generators solve: it's mechanical,
error-prone, and the failures are subtle (missing `activationEvents`,
malformed `contributes.debuggers`, wrong `main` entry).

This generator eliminates that boilerplate.  Run it, get an extension
that compiles via `tsc`, packages with `vsce`, and installs into
VS Code with `code --install-extension`.

## Stack position

```
              language author runs the generator once
                              │
                              ▼
   ┌─────────────────────────────────────────────────────┐
   │   <lang>-vscode/    (this generator's output)       │
   │   ├── activate() wires LSP + DAP                    │
   │   ▼                                                 │
   │  vscode-languageclient   ◀── stdio LSP ──▶ <lang>-lsp-server │
   │  vscode.debug API        ◀── stdio DAP ──▶ <lang>-dap        │
   └─────────────────────────────────────────────────────┘
                              │
                              ▼
              user opens a .<ext> file in VS Code
              and gets language intelligence + debugger.
```

## Usage

### Spec-driven flow (recommended)

Each language built on the LANG-VM pipeline ships a `<lang>-spec-dump`
binary in its parser crate (e.g. `twig-spec-dump`).  That binary pulls
the build-time-compiled lexer + parser grammars straight out of the
crates and prints a `LanguageSpec` JSON document to stdout.  The
generator consumes it:

```bash
twig-spec-dump > twig.spec.json

vscode-lang-extension-generator \
  --language-spec twig.spec.json \
  --lsp-binary twig-lsp-server \
  --dap-binary twig-dap \
  --output-dir code/packages/typescript/twig-vscode
```

That's it.  No manual keyword listing, no manual file-extension
declarations — they all come from the lexer's compiled view of the
language.  When the language adds a new keyword, regenerating
`twig.spec.json` and re-running the generator picks it up
automatically.

### Why this works without source files

The canonical `code/grammars/<lang>.tokens` and `code/grammars/<lang>.grammar`
files are **build-time-only** artifacts.  At runtime, the language's
lexer rlib has the parsed `TokenGrammar` baked in as struct literals
(via `<lang>-lexer/build.rs` calling `grammar_tools::compiler`); the
parser rlib does the same for `ParserGrammar`.  `<lang>-spec-dump`
reads those compiled artifacts and serialises them.

Single source of truth: whatever the lexer recognises is what the
editor sees.

### Manual flag flow (no spec)

For experiments or for languages that don't yet have a `*-spec-dump`,
all the metadata can still be passed via flags:

```bash
vscode-lang-extension-generator \
  --language-id twig \
  --language-name Twig \
  --file-extensions .twig,.tw \
  --lsp-binary twig-lsp-server \
  --dap-binary twig-dap \
  --output-dir <path> \
  --line-comment ";" \
  --keywords "define,if,let,lambda"
```

### Flags

| Flag | Required | Description |
|---|---|---|
| `--language-spec` | spec-driven | Path to a `*-spec-dump` JSON file.  When set, drives all language metadata. |
| `--language-id` | when no spec | Internal slug (e.g. `twig`).  Matches `^[a-z][a-z0-9-]*$`. |
| `--language-name` | when no spec | Display name (e.g. `Twig`). |
| `--file-extensions` | when no spec | Comma-separated, leading dots: `.twig,.tw`. |
| `--output-dir` | yes | Directory to create.  Errors if non-empty. |
| `--lsp-binary` | * | Name of LSP server binary (e.g. `twig-lsp-server`). |
| `--dap-binary` | * | Name of DAP adapter binary (e.g. `twig-dap`). |
| `--line-comment` | | Override the spec's line-comment marker. |
| `--block-comment-start` | | Override the spec's block-comment open. |
| `--block-comment-end` | | Override the spec's block-comment close. |
| `--keywords` | | Override the spec's keyword list (comma-separated). |
| `--description` | | Defaults to autogenerated. |
| `--ext-version` | | Defaults to `0.1.0`. |

\* At least one of `--lsp-binary` or `--dap-binary` must be provided —
otherwise the extension would do nothing.

When `--language-spec` is set, every other CLI flag is an **optional
override** — explicit flag values win, otherwise the spec value is
used.

## Output layout

```
<output-dir>/
├── package.json              # contributes.languages/grammars/debuggers/configuration
├── tsconfig.json
├── .vscodeignore
├── README.md
├── CHANGELOG.md
├── BUILD                     # npm install + tsc
├── BUILD_windows
├── language-configuration.json
├── src/
│   ├── extension.ts          # always
│   ├── lsp.ts                # iff --lsp-binary
│   └── dap.ts                # iff --dap-binary
├── syntaxes/
│   └── <lang>.tmLanguage.json    # iff --keywords
└── examples/
    └── sample<first-extension>
```

The generated `extension.ts` reads `<lang-id>.serverPath` and
`<lang-id>.adapterPath` from VS Code settings, falling back to PATH
lookups by binary name — the same convention as rust-analyzer and
Pylance.

## What is *not* generated

- The LSP server / DAP adapter binaries themselves.  Install those
  separately (`cargo install`, `brew install`, prebuilt download).
- Rich syntax highlighting beyond the keyword list.  Hand-author or
  generate a richer grammar in a separate package.
- Marketplace metadata (publisher, icon, repository URL).  Edit
  `package.json` after generation.

## Determinism

Re-running the generator with identical inputs produces byte-identical
output.  Object keys are emitted in a fixed order, no timestamps appear
anywhere in the output.

## Testing

```bash
./BUILD                       # install transitive deps, run tests + coverage
```

68 tests covering:

- Input validation (kebab-case ids, file extensions, capability gating)
- Each builder function (`buildPackageJson`, `buildExtensionTs`,
  `buildLspTs`, `buildDapTs`, `buildLanguageConfiguration`,
  `buildTextMateGrammar`)
- Filesystem orchestration (`generate`)
- CLI parsing and error paths
- A smoke test that runs `tsc --noEmit` against the generated source

Coverage: ~98%.
