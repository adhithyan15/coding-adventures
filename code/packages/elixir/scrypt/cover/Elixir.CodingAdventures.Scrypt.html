2 @moduledoc """
3 scrypt — Memory-Hard Password-Based Key Derivation Function (RFC 7914).
4
5 ## What Is scrypt?
6
7 scrypt is a password hashing function designed by Colin Percival in 2009. It
8 was specifically engineered to be **memory-hard**: computing it requires not
9 just CPU time, but also a large amount of RAM. This makes it very expensive
10 to attack using ASICs (custom chips) or FPGAs, which can parallelize SHA-256
11 cheaply but cannot cheaply parallelize access to large amounts of memory.
12
13 Real-world use cases:
14 - Litecoin (the cryptocurrency) uses scrypt as its proof-of-work hash
15 - OpenBSD's `bcryptpbkdf` borrows ideas from scrypt
16 - macOS File Vault key derivation
17 - Tarsnap backup tool (scrypt's original application)
18
19 ## Why Not PBKDF2?
20
21 PBKDF2 (and bcrypt) are compute-bound — every iteration is purely CPU work.
22 Modern GPUs can perform billions of SHA-256 hashes per second at low cost,
23 so PBKDF2 passwords can be brute-forced efficiently on GPU clusters.
24
25 scrypt changes the economics: each candidate password requires reading/writing
26 a large block of memory (`N * 128 * r` bytes). Memory bandwidth is the
27 bottleneck, and memory cannot be parallelized like compute. Even expensive GPU
28 clusters lose their advantage when memory bandwidth per thread is the limit.
29
30 ## Parameters
31
32 | Parameter | Meaning | RFC 7914 default |
33 |-----------|----------------------------------------------|-----------------|
34 | `n` | CPU/memory cost (must be power of 2, ≥ 2) | 16384 |
35 | `r` | Block size factor (typically 8) | 8 |
36 | `p` | Parallelization factor (typically 1–4) | 1 |
37 | `dk_len` | Output length in bytes | 32 |
38
39 Memory usage = `128 * r * n` bytes (e.g. r=8, n=16384 → 16 MiB).
40
41 ## Algorithm Overview (RFC 7914 §3)
42
43 scrypt is built from three layers:
44
45 ```
46 scrypt(password, salt, N, r, p, dk_len)
47 │
48 ├─ Step 1: PBKDF2-HMAC-SHA256(password, salt, 1, p*128*r)
49 │ Derive p independent 128*r-byte blocks.
50 │
51 ├─ Step 2: For each block, run ROMix(block, N)
52 │ ROMix fills an N-entry lookup table (the "ROM"),
53 │ then makes N pseudo-random lookups into it.
54 │ This is the memory-hard step.
55 │
56 └─ Step 3: PBKDF2-HMAC-SHA256(password, mixed_blocks, 1, dk_len)
57 Finalize the output key.
58 ```
59
60 ### Salsa20/8 Core
61
62 The innermost primitive is a variant of the Salsa20 stream cipher, using
63 only 8 rounds instead of the full 20 (hence "Salsa20/8"). It operates on
64 64-byte (512-bit) blocks interpreted as 16 little-endian uint32 words.
65
66 Each "double round" applies a quarter-round function to columns, then rows.
67 After 4 double-rounds (= 8 total rounds), the final state is added word-by-word
68 to the initial state (modulo 2^32).
69
70 ### BlockMix
71
72 BlockMix operates on a 128*r-byte block treated as 2r sequential 64-byte
73 chunks. It XORs and feeds each chunk through Salsa20/8 sequentially, then
74 deinterleaves: even-indexed outputs come first, odd-indexed outputs second.
75 This deinterleaving is what gives scrypt its memory-access pattern.
76
77 ### ROMix
78
79 ROMix builds a random-access memory table of N BlockMix outputs (V[0]..V[N-1]),
80 then makes N pseudo-random lookups: each step XORs the current block with a
81 table entry chosen by the last 8 bytes of the current block (Integerify),
82 then runs BlockMix again. Because the lookup indices are pseudo-random and
83 depend on previous outputs, the full table must be kept in memory — it cannot
84 be recomputed on-the-fly without revisiting earlier entries.
85
86 ## Example
87
88 iex> CodingAdventures.Scrypt.scrypt_hex("", "", 16, 1, 1, 64)
89 "77d6576238657b203b19ca42c18a0497f16b4844e3074ae8dfdffa3fede21442fcd0069ded0948f8326a753a0fc81f17e8d3e0fb2e0d3628cf35e20c38d18906"
90
91 ## Security Notes
92
93 - Use N ≥ 16384, r = 8, p = 1 for interactive logins (2013 recommendation).
94 - Use N ≥ 1048576, r = 8, p = 1 for sensitive data at rest.
95 - For new systems, Argon2id (RFC 9106) is preferred — it won the Password
96 Hashing Competition and has better theoretical properties. But scrypt
97 remains widely supported and is a solid choice.
98 - Never use the same (password, salt) pair for two different purposes.
