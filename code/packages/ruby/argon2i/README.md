# coding_adventures_argon2i

A pure-Ruby, from-scratch implementation of **Argon2i** (RFC 9106) —
data-independent memory-hard password hashing.

## What is Argon2i?

Argon2i uses **data-independent** addressing: reference indices come
from a stream of pseudo-random values generated by running the Argon2
compression function on an input block derived purely from public
parameters (pass number, lane, slice, counter). Memory access
patterns leak nothing about the password, making Argon2i resistant
to side-channel attacks.

The trade-off is weaker resistance to time-memory trade-off attacks
on GPUs/ASICs. For general password hashing, prefer
[`argon2id`](../argon2id/); use Argon2i only when side-channel
resistance is the dominant concern.

See the spec at [code/specs/KD03-argon2.md](../../../specs/KD03-argon2.md).

## Usage

```ruby
require "coding_adventures_argon2i"

tag = CodingAdventures::Argon2i.argon2i("password", "somesalt", 3, 64, 1, 32)
hex = CodingAdventures::Argon2i.argon2i_hex("password", "somesalt", 3, 64, 1, 32)
```

### Keyed / authenticated data

```ruby
tag = CodingAdventures::Argon2i.argon2i(
  password, salt, 3, 64, 1, 32,
  key: secret,
  associated_data: "user:alice"
)
```

## API

| Method | Returns |
| -- | -- |
| `Argon2i.argon2i(password, salt, t, m, p, T, **opts)` | `String` (binary, `T` bytes) |
| `Argon2i.argon2i_hex(password, salt, t, m, p, T, **opts)` | `String` (lowercase hex) |

Parameters follow RFC 9106 §3.1.

## Where this fits in the stack

- **Dependencies:** [`coding_adventures_blake2b`](../blake2b/) (H0 and H' extender).

## Security notes

- **Trust boundary on `memory_cost` and `tag_length`.** RFC 9106 permits
  both up to `2^32 - 1`, which translates to multi-TiB allocations.
  Clamp caller-controlled values at the application layer.
- **Verify in constant time.** Use `OpenSSL.fixed_length_secure_compare`
  rather than `==` when comparing a stored tag against a freshly
  computed one.

## Running the tests

```bash
cd code/packages/ruby/argon2i
bundle install
bundle exec rake test
```

Tests include the canonical RFC 9106 §5.2 gold-standard vector, plus
18 parameter-edge tests (19 total) covering validation, determinism,
binding to key/AD, tag-length variants, and multi-lane / multi-pass
parameters.
